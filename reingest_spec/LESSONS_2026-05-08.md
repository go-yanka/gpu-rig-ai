# Lessons learned — 2026-05-08 (CP-3 push for 95% trust)

Permanent record. Read this before re-attempting any G1 / G2 / G3 / G4 work on cbic_v2.

## L1. SPEC §1 is the bar. Don't softify with "adjusted recall."

The frozen SPEC says: "≥95% trust across four gates: Accuracy [G1], Reasoning [G2], Evidence [G3], Refusal [G4]" — recall@10 on 350-400 query gold set, no carve-out clause. The historical "adjusted recall" methodology (excluding queries probing carve-out doc_ids) is selection bias that **independent auditors will reject.** Always run gates on the full SPEC gold and report raw numbers. If raw <0.95, fix the corpus / retrieval / chunker; do not redefine the metric.

**Mistake made today:** I dismissed G1 0.5474 as "informational" by reading RG's "trustable reasoning" emphasis as "G1 doesn't count." SPEC says all four count.

## L2. HyDE alone HURTS recall on long-scenario business queries

Before today, HyDE was disabled (`USE_HYDE=0` in API systemd env). Turning it on for the 380-query gold dropped G1 from 0.7585 → 0.7542. Why: qwen3 generates generic hypothetical answers ("countervailing duty notification on copper") that pull retrieval *away* from the specific notification number expected by gold. Hybrid (dense+sparse) without HyDE is strictly better for these queries.

**Standing rule:** measure HyDE on/off A/B before assuming it helps. For specific-document queries with company names, dates, notification numbers, HyDE often hurts.

## L3. DENSE_ONLY=1 is BOTH ingest-time AND query-time

The env var was set on `cbic-rag-api.service` to mirror the ingest setting. **At query time it disables sparse retrieval entirely — even when sparse vectors exist in Qdrant.** Two flips needed to enable hybrid: (a) sparse vectors must be present in chunks, (b) DENSE_ONLY=0 in API env. Codified `/etc/systemd/system/cbic-rag-api.service.d/hybrid.conf` (mirrored to `D:/_gpu_rig_ai/reingest_spec/systemd/...`).

## L4. Sparse vectors can be backfilled WITHOUT re-ingesting dense

`embed_sparse_batch()` from `cbic_rag/embedder.py` is CPU-only via fastembed BM25, ~3ms/chunk. Backfill 38K chunks in ~100s via Qdrant scroll + `update_vectors`. Far faster than the codified preflight rule's worry ("phase4 sparse on CPU is slow"). Permanent script: `reingest_spec/scripts/add_sparse_v2.py` (uses Qdrant point IDs from scroll, NOT reconstructed hashes — Python `hash()` is non-deterministic across processes).

**Standing rule:** if sparse missing from Qdrant, run `add_sparse_v2.py` rather than re-ingesting.

## L5. ingest_v2.py preflight has stale GPU rules

Line 89-90: `EMBED_GPUS_DEFAULT = "1,3,4,5,6"` and `EMBED_GPUS_FORBIDDEN = {"0", "2"}` exclude GPU 0 and GPU 3, while project memory + run_batch_loop use 6-card pool `{0,1,3,4,5,6}`. Inconsistency. For now, ingest with the stale 5-card pool; **TODO** reconcile by removing GPU 0 from `EMBED_GPUS_FORBIDDEN` (we have empirical data that GPU 0 works fine for embedding alongside the bge-reranker — it's an RX 6700 XT 12GB, plenty of headroom).

## L6. G1 evaluator's per_item is in COMPLETION order with workers>1

When `--workers N>1`, `gate_g1_recall.py` collects per-query results via `as_completed(futures)`, so `per_item[i]` does NOT correspond to `gold[i]`. Mapping by index is wrong. To diagnose specific misses:
- Either run with `--workers 1` (deterministic, slower)
- Or compare hit rates aggregate, not per-position

I wasted ~20 min today on a "concurrency contamination" hypothesis that was actually just per_item-vs-gold ordering mismatch.

## L7. D-1 cluster `linked_doc_ids` is incomplete by chunker design

Chunker only links `doc_ids` whose chunks it actually processed (via the dedupe collision path). For shared-PDF clusters where build_batch never picked all members, the cluster relationship isn't fully recorded. The G1 evaluator's Defect-D fallback (`linked_doc_ids` substring match) misses these.

**Permanent fix codified:** `reingest_spec/scripts/backfill_linked_doc_ids.py` — for each `sha256_en` cluster of size >1, sets every cluster member chunk's `linked_doc_ids` to the full cluster minus self. Idempotent. Run after any ingest that adds new docs to existing clusters.

**Standing rule:** chunker is "doc-by-doc local"; cluster-wide invariants must be backfilled separately.

## L8. 24 of 25 sampled G1 misses had expected doc NOT in dense top-50

Categorical: dense embedding (BGE-M3 1024d) **cannot bridge** long-scenario business queries to specific tax notifications/circulars. The semantic similarity between "Hindalco-Bharat Copper Works LLP, a manufacturer of copper wires" and "Notification 1002110 — countervailing duty 12% on copper wire imports from country X" is genuinely low. Only sparse (BM25) keyword matching on "Hindalco" / specific notification numbers / acronyms can find these.

**Standing rule:** for tax/legal RAG with long business-scenario queries, hybrid (dense+sparse) is mandatory, not optional.

## L9. Independent verification means SPEC-as-written, not project conventions

RG explicitly stated: "all the audits will be verified again using independent agents, so no fake pass." This means:
- Run gates on full SPEC gold (380 queries), no carve-out filtering
- Report raw recall, not "adjusted"
- Document carve-outs as backlog items (not as gate adjustments)
- Any rule deviation (e.g. "we excluded D-1 misses because of structural chunker limit") will be flagged by independent auditors

**Standing rule:** if you find yourself adjusting denominators / numerators to reach 95%, stop. Fix the underlying corpus or retrieval issue.

## L13. Per-batch verification ≠ corpus-wide verification (silent stragglers bug)

**The problem:** `run_batch_loop.sh` invokes `ingest_v2.py --phase all --doc-ids "$DOC_LIST"`. The `--doc-ids` filter applies to ALL phases including phase3_4_5. If phase3_4_5 errors partway through a batch (Qdrant timeout, embed batch failure, etc.), the failed chunks stay at `upserted=0` in the v2 manifest. The next batch only operates on its own `--doc-ids` and never re-attempts the stragglers. They leak silently.

**The blind spots:**
- `post_batch_lint.py` checks "expected docs have ≥1 chunk" but is scoped to current batch's CSV. Stragglers from other batches are out of scope.
- `phase3_4_5 RECONCILE` check verifies "scoped docs have expected==upserted" — also scoped. Happily passes for the current batch even when 1,098 OTHER doc_ids' chunks are stuck.

**The damage today (2026-05-08):**
- 3,755 canonical chunks across **1,098 distinct doc_ids** stuck at `upserted=0`
- ~70 of these are in the G1 gold set → directly cost ≥30% of our recall
- Took ~5 hours of CP-3 push to surface

**Permanent fixes to codify (TODO):**
1. **Corpus-wide drain at end of `run_batch_loop.sh`**: after the last batch, run `ingest_v2.py --phase phase3_4_5` with NO `--doc-ids` filter. Picks up every pending upserted=0 across the manifest.
2. **Cross-batch audit in `post_batch_lint.py`**: query manifest for `SELECT COUNT(*) FROM chunks WHERE is_canonical=1 AND upserted=0` — if >0, emit P1 finding. Doesn't HALT (some staggering is OK during loop) but surfaces the count.
3. **Drain check before any CP gate panel fires**: trust gates can't measure what isn't in Qdrant.

**Standing rule:** per-batch determinism is necessary but insufficient. Always assert corpus-wide invariants between batches. If a manifest column tracks state ("upserted", "phase2_done"), there must be a corpus-wide query that ensures `0 or non-zero is intentional`.

## L12. Retrieval gap diagnosis methodology

When a gate fails, do NOT blindly tune θ or expand k. The correct sequence:
1. Run gate, get hit count + miss count
2. Sample 20-25 misses, probe `/retrieve` with `k=50` for each
3. Bucket misses: (a) in top-10 of retrieve but evaluator misses → evaluator bug; (b) at rank 11-50 → reranker θ issue; (c) rank 50+ or not in top-50 → embedding/chunker issue, sparse needed; (d) expected_doc_id not in any chunk → corpus coverage issue
4. Fix the dominant bucket first

This methodology codified today. Saved hours of misdirected tuning.
