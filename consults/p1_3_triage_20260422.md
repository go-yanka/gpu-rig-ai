## Hypothesis — WHY two-pass regresses

**Primary cause: pass-1 extraction under-recall + conservative refusal gate.**

1. Pass-1 prompt asks the LLM to return JSON of verbatim spans that exactly match the retrieved chunks. For many questions the answer is compositional (requires stitching across chunks, or paraphrasing a rule) and no single verbatim span captures it.
2. When pass-1 returns empty / all-rejected, the pipeline refuses outright instead of falling back to a single-shot generation. **This is the 54-item cliff.**
3. Even when pass-1 returns spans, pass-2 synthesis operates only on those spans (or spans + chunks in Eval-B), which truncates useful context the single-shot path sees. Pass-2 prose is therefore more hedged and lower-scoring — that's the 38 wrong-content drops.
4. Chunk re-feed (Eval-B) did not rescue pass 2 because pass-2's system prompt is already conservative; giving it more chunks without loosening the "must be grounded in pass-1 extractions" constraint changes little.

## Is the fix obvious enough to retry?

**No — needs redesign.** The architectural mistake is treating empty pass-1 as a hard refusal. Two reasonable redesigns:

- **Option R1 — Soft fallback.** If pass-1 yields zero validator-accepted spans, fall back to the P1.2 single-shot path rather than emit the canned refusal. Preserves two-pass on easy items, avoids the cliff on compositional items. Simplest change.
- **Option R2 — Loosen validator.** Allow pass-1 to return paraphrased spans with chunk attribution (not strict verbatim). Higher recall but risks hallucination.

Neither is ready to deploy today. **P1.3 stays parked** — flag OFF as documented in morning status. If resumed, prefer R1 (soft fallback) and re-run Eval-A only before Eval-B.
