# P2.4 NER-Filter Blocker — 2026-04-22

**Status:** cannot implement as designed. Requires P2.3 corpus enrichment first.

## What the design assumed

The design (`p2_4_ner_filter_design.md`) assumes chunk payloads carry arrays of *cited* entities:

- `sections: ["16", "16(2)", "16(2)(c)"]`
- `rules: ["36", "36(4)"]`
- `notifications: ["12/2017-CT(R)"]`
- `hsn: ["9983","998311"]`
- etc.

These arrays would let a query like "Section 16(2)(c) CGST" filter to chunks **that cite** Section 16(2)(c), regardless of which document the chunk is from.

## What the live collection actually has

Audited `cbic_v1` on rig (108,802 points). Payload schema has only **scalar** fields:

- `section_ref` (keyword, 17% populated): represents *the chunk's own internal section number within its source document*, not cited sections. Top values: `18(2)(b)`, `18(2)(iii)`, `18(2)`, `18(12)` — almost entirely from one document's internal clauses, not CGST/IGST/UTGST act sections.
- `doc_number` (keyword, 82% populated): the chunk's *source* notification/circular number, e.g. `3/2017-Centra`, `50/2017-Cus`, `22/2022-Custom`. Note truncation (`Centra`, `CUSTOM`, `Custom` variants) — not canonicalized.
- No `sections`, `rules`, `notifications`, `hsn`, `sac`, `forms`, `acts` arrays.

Consequence: a `must` filter on `section_ref=16` would return **zero or near-zero** chunks for any gold-set section-lookup query, because the chunks discussing CGST Section 16 don't carry `section_ref=16` — they carry the section_ref of whatever clause of whatever circular/notification contains that discussion (often empty or unrelated like `18(2)(b)`).

A `must` filter on `doc_number=50/2017-Cus` would return only chunks **from** that notification's text, which is sometimes useful (for "what does notification 50/2017-Cus exempt") but also ignores chunks from circulars/rulings *citing* that notification.

## Fallback options considered

1. **Doc-number-only filter.** Detect `<num>/<year>-(Cus|CT|IT|ST|CE)` in query, filter `doc_number` via `MatchAny` on common canonical variants (e.g. `50/2017-Cus`, `50/2017-Custom`, `50/2017-CUSTOM`). Thin win: helps maybe 3-5 gold items (`customs_notif_001`, `gst_notif_002`, `gst_notif_003`), but misses all section/rule queries — which are the common case.
2. **Soft-boost.** Instead of `must` filter, rank up matching chunks. Doesn't need schema changes. But requires re-scoring in retriever with custom logic, and gains are likely small since dense+BM25 already rank notification-number queries reasonably.
3. **Full P2.3 enrichment.** Re-extract citations from chunk text via regex, write back to Qdrant payloads, add keyword indexes, then implement P2.4 on top. This is the right path but takes hours of re-ingest and is out of day-shift scope.

## Decision

**Skip P2.4 in this shift.** Document as blocker. Preferred follow-up: schedule P2.3 overnight (regex-extract citation arrays, update payloads, add indexes) and retry P2.4 against enriched collection.

Low-cost alternative to try next shift if wanted: option 1 above (doc-number-only filter), scoped to the ~5 gold items it plausibly helps, guarded by `NER_FILTER_ENABLED=1`. Won't move the needle meaningfully but validates the wire-in pattern.
